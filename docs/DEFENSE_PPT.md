## 第 5 页 · 系统架构（1.5 分钟）

**内容**:
- 多进程架构图（从 TECHNICAL_REPORT.md §2.1 复制 Mermaid 图）
- 8 条队列通信：frame / display / command / adjustment / gallery / draw / animation / result
- Consumer 14 步帧循环：取帧 → AI 调度 → 人脸检测 → 编辑/画廊/绘制/动画 队列处理 → 渲染 → 推帧
- 关键设计决策：Mixin 组合模式 · typed dataclass 协议 · StickerRegistry 域模型

**演讲要点**: "为什么要用多进程？摄像头采集、AI 推理、UI 渲染三者不能互相阻塞。8 条队列各司其职，所有消息用 typed dataclass 定义在 protocol.py 里——这是架构层面的'合同'，也是质量保证的基石。"

---

## 第 6 页 · AI 生成管线（1 分钟）

**内容**:
- 端到端流程图：用户输入 → DeepSeek Agent（多轮对话 + JSON 结构化输出）→ ComfyUI 工作流注入（Prompt/LoRA/Seed）→ SDXL + Layer Diffusion → 透明 PNG
- 降级策略：无 API Key 或无 GPU → 关键词匹配（9 区域 60 关键词）+ Mock 模式
- 风格预设：4 种内置风格（像素/矢量/卡通/半写实），LoRA 动态切换

**视觉**: 流程图 + 4 种风格生成效果对比

**演讲要点**: "AI 管线的核心挑战是可靠性——外部服务随时可能挂。我们做了三层防护：DeepSeek API 失败自动降级关键词 → ComfyUI 超时重试 → Mock 模式保证 UI 至少能跑。"

**案例深讲 — E1 "修改引入回归"**:
- 初期状态：0 测试，每次改代码提心吊胆（S=7, O=9, D=9 → RPN=567）

**案例深讲 — T1 "ComfyUI 无响应"**:

- 应对：超时重试（120s）+ 自动降级 Mock + 最近 ComfyUI 健康检查

**演讲要点**: "RPN 公式（S×O×D）给了我们优先级排序的依据。E1 的 RPN 高达 567，我们在 P1 就开始还测试债。T1 只有 378，但对外部服务我们只能缓解不能消除——这就是为什么 Mock 模式作为应急方案很重要。"
